# Polymarket pool participant allocation accounting

## 触发场景

当主人要求记录“谁参加了下注 / 每个人本金 / 利润占比 / 池子占比 / 每个人每场或每个市场下注多少”时使用。

## 核心原则

1. 先确定账面总额口径：本金 + 已实现利润分配，不把未实现浮盈亏默认并入本金，除非主人明确要求。
2. 用户纠正本金或转让份额时，立即重算余额和占比；不要沿用旧金额。
3. 利润分配和本金占比是两套口径：
   - 利润分配：按主人指定比例分某一场已实现利润。
   - 池子占比：按当前每个人余额 / 当前池子总额。
4. account1/account2 的实际下注按“市场成本”分摊，而不是只按比赛合计；例如 USA vs Paraguay 必须拆成 `USA 胜 Yes` 与 `USA Under 2.5`。
5. 不同市场的 shares/size 不能相加；可以合计 USDC 金额。

## 推荐计算流程

```text
1. 读取/确认参与人余额：本金 + 已实现利润分配 - 转让/提现 + 补入。
2. 计算池子总额：sum(参与人余额)。
3. 计算每人占比：个人余额 / 池子总额。
4. 读取当前账户实际持仓成本：positions.initialValue 或已核验订单成本；按 account1/account2 分开保存，再按需要合并。
5. 按每个独立市场分摊：个人该市场金额 = 市场成本 * 个人占比。
6. 按比赛小计：同一比赛多个市场的个人金额加总。
7. 输出：余额/占比、每个市场分摊、每场小计、总分摊。
8. 每场比赛结束后，读取该场各 market 的最终已实现回收/结算金额，计算场级盈亏：场利润 = 该场回收金额 - 该场成本。
9. 按比赛结束时的池子占比分配该场盈亏：个人场盈亏 = 场利润 * 个人占比；若主人指定该场特殊利润比例，则用特殊比例覆盖池子占比。
10. 更新每人余额：新余额 = 旧余额 + 个人场盈亏；再重算新池子总额和新占比，作为下一场/下一轮分摊基础。
```

## 输出模板

### 赛前/未平仓分摊

```text
池子总额：XXXX USD

参与人：
- A：余额 ...，占比 ...%
- B：余额 ...，占比 ...%

account1 当前下注：
- USA vs Paraguay / USA 胜 Yes：600 USDC
- USA vs Paraguay / USA Under 2.5：399.9964 USDC

按人 × 市场：
A：
- USA 胜：...
- USA Under 2.5：...
- 本场小计：...
- 总分摊：...
```

### 赛后结算/余额更新

```text
比赛：USA vs Paraguay

市场结算：
- USA 胜 Yes：成本 600，回收/结算 X，盈亏 X-600
- USA Under 2.5：成本 399.9964，回收/结算 Y，盈亏 Y-399.9964
- 本场总成本：999.9964
- 本场总回收：X+Y
- 本场总盈亏：X+Y-999.9964

按人分配盈亏：
- A：旧余额 ...，占比 ...%，本场盈亏 ...，新余额 ...
- B：旧余额 ...，占比 ...%，本场盈亏 ...，新余额 ...

新池子总额：...
新占比：...
```

若该场有主人指定特殊利润分配（例如 Mexico 第一场利润按 粟粟40%、ray20%、Steve40%），必须在“按人分配盈亏”前单独标注，并用特殊比例覆盖池子占比。

## 本次会话沉淀的例子

用户纠正：

```text
Steve 本金从 7486 改为 7427；Steve 让出 727 给小韩；
第一场 Mexico 利润按 粟粟40%、ray20%、Steve40% 分配。
```

正确处理：

```text
Steve 当前余额 = 7427 - 727 + 第一场利润*40%
小韩余额 = 727
粟粟余额 = 原本金/参与额 + 第一场利润*40%
ray余额 = 原本金/参与额 + 第一场利润*20%
```

然后用新的余额占比去分摊 account1 当前所有未平仓市场。

## 常见坑

❌ 只按比赛给每人分摊，不拆具体市场。
✅ 按 `比赛 / market / outcome` 粒度分摊，例如 `USA 胜 Yes` 与 `USA Under 2.5` 分开。

❌ 把已实现利润占比当成池子长期占比。
✅ 利润分配只影响余额；池子占比每次按最新余额重算。

❌ 将 shares/size 汇总到个人总份额。
✅ 只汇总金额，shares 必须按独立市场保留。

❌ 私聊中记录明细但不落盘。
✅ 保存到私有目录或项目账本，并给出文件路径；不要把私钥/API secret 写入账本。
